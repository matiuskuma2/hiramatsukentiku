# 平松建築 概算原価管理システム - プロジェクト概要

## 1. プロジェクトの目的

### 最優先目的
**原価計算ミスをなくし、粗利ブレを減らすこと。**

### 背景
- YouTube経由でのリード獲得は好調だが、受注後に利益が崩れる構造になっている
- 見積もり精度の低さが粗利を圧迫し、年間数千万円規模の機会損失が発生
- 粗利が不安定なため、事業拡大に必要な人材投資が進まない
- 現状はスプレッドシートで原価計算を運用しており、人的ミス・条件漏れが頻発

### 解決すべき構造
```
集客OK → 受注後に利益が崩れる → 採用できない → キャパ不足 → 成長停止
```

### このシステムで解決すること
```
見積精度向上 → 粗利安定 → 採用加速 → 受注拡大 → 成長
```

---

## 2. システムの位置づけ

### 今回作るもの
**社内の案件別概算原価管理システム**

- 顧客向け見積書を作るシステムではない（将来的には連携）
- 社内で原価を正しく積み上げ、粗利を守るための基盤

### 商談フロー上の位置づけ
```
[Phase 0] 商談：ライフプラン + ざっくり見積もり（資金計画比較表）
    ↓
[Phase 1] 社内概算原価計算 ← ★今回のメイン
    ↓
[Phase 2] 精緻見積・契約前見積
    ↓
[Phase 3] 実行予算・着地原価
```

### 今回の範囲
- Phase 1（社内概算原価計算）に集中
- Phase 0（商談資金計画）は案件に紐づけるが、本体ロジックとは分離
- Phase 2以降は将来拡張

## 4. 原価シート分析から見えた設計原則

### 4-1. この原価シートは「計算機」ではなく「判断支援システム」

金額が決まる仕組みは以下の**12パターン**が混在している：

| # | 計算方式 | 代表例 | コード |
|---|---------|--------|--------|
| 1 | 固定額 | レッカー、外注監査、地盤保証 | `fixed_amount` |
| 2 | 坪単価 | 金物、現場管理費、電気設備 | `per_tsubo` |
| 3 | 面積単価 | 断熱材、外壁、屋根、美装 | `per_m2` |
| 4 | メートル単価 | 深基礎、幕板、軒樋 | `per_meter` |
| 5 | 個数単価 | 通線口、養生テープ | `per_piece` |
| 6 | 面積帯/坪数帯ルックアップ | 基礎面積帯、瑕疵担保保険 | `range_lookup` |
| 7 | ラインナップ別固定額 | 木工事(MOKU)、設計業務 | `lineup_fixed` |
| 8 | ルール表参照で数量算出 | WB部材、大工人工、断熱等級切替 | `rule_lookup` |
| 9 | 都度見積 | 土工事、地盤改良、家具製造 | `manual_quote` |
| 10 | 商品選択積み上げ | サッシ、内装建具、畳、太陽光 | `product_selection` |
| 11 | 標準セット＋差額管理 | 産廃ボックス | `package_with_delta` |
| 12 | しきい値付き加算 | 水道70m超、左官10m2以下一式 | `threshold_surcharge` |

> **注**: v2改訂で `per_piece`（個数単価）を追加し、10→12パターンに拡張。詳細は 07_CROSS_REVIEW_AND_RESOLUTIONS.md C-02 参照。

### 4-2. 「概算」の思想を尊重する

現行シートは意図的にラフな数字を使っている：
- 屋根面積に軒・下屋根も含む（少し多めに見る）
- 床面積に土間部分も含む
- カット代は含まない
- 過去実績から標準化した坪単価を使う

**→ システムでもこの「概算モード」の考え方を残す。**
いきなり厳密積算に寄せない。

### 4-3. ミスの本質は計算ミスより「条件漏れ」

分析の結果、ズレの主因は：
1. 行政・自治体条件の確認漏れ（水道工事、浄化槽）
2. 図面部位の数量拾い漏れ（タイル、内装仕上）
3. 現場条件の見落とし（左官、屋根形状例外）
4. エリア条件の判定漏れ（美装、水道、産廃）
5. 仕様未選択のまま進む（外壁材、サッシ）

**→ 計算式移植だけでなく、確認フロー・警告・要確認フラグが必須。**

### 4-4. 工種間連動がある

- 外壁で焼杉採用 → 木工事に加工費追加
- ラインナップ選択 → 木工事/設備/照明のロジック切替
- 建築地選択 → 産廃・美装・水道の地域判定

**→ 明細単独で閉じず、他工種参照の仕組みが必要。**

## 5. 開発フェーズ

### Phase 1（今回）：現行シート完全再現
- スプレッドシートの項目・計算ロジック・確認フローを忠実に再現
- 同じ条件 → 同じ工種 → 同じ金額が出る状態にする
- 手修正・確認フラグ・更新履歴も再現

### Phase 2：運用改善UI
- 工種の見やすい整理
- 自動追加ルールの分かりやすい表示
- 漏れやすい条件に対する警告UI
- 差額理由入力の強化

### Phase 3：分析・改善
- 見積原価 vs 実績原価の差異分析
- どの工種でズレるか
- 誰の案件でズレるか
- 初回提示額との差

### Phase 4：外部連携
- 商談資金計画との統合
- ANDPAD連携
- 発注書発行
- 帳票出力

## 6. 重要な設計判断

### 6-1. 商談用ざっくり見積もりとの分離
- 案件マスタは共通
- 見積レイヤーは分ける（商談概算 / 社内概算原価 / 契約前見積 / 実行予算）
- 同じ画面・同じロジックで扱わない

### 6-2. 粗利率は工種別に分けられるようにする
- 標準粗利率と太陽光粗利率は最低限分離（現行シート準拠）
- 将来的にはオプション粗利率も別管理

### 6-3. 単価スナップショット
- 案件作成時点のマスタ単価をスナップショット化
- 後日の単価変更で過去案件の数字が変わらないようにする

### 6-4. AIは「提案」、最終判断は「人間」
- AIに原価を決めさせない
- AIは入力補助・チェック補助・資料読取に使う
- AI出力は必ず確認画面を通す

## 7. KPI（将来の効果測定用）

| 指標 | 内容 |
|------|------|
| 見積粗利率 vs 実績粗利率の差異 | 物件ごと・担当者ごと・商品タイプごと |
| 追加原価の発生理由 | 分類して集計 |
| 見積修正回数 | 案件あたり |
| 契約後増額件数 | 件数と金額 |
| 初回提示額 → 最終契約額の増減率 | 営業精度の改善度 |
| 条件漏れ検出件数 | システム導入前後で比較 |

## 8. 用語定義

| 用語 | 定義 |
|------|------|
| ラインナップ | 商品シリーズ（SHIN, RIN, MOKU_OOYANE, MOKU_HIRAYA, MOKU_ROKU） |
| 断熱等級 | 断熱性能等級（5, 6） |
| 工種 | 工事の種類（基礎工事、木工事、断熱材 等）。**全37工種**（標準35 + 太陽光 + オプション） |
| 明細 | 工種の中の個別項目（深基礎、上棟加算 等） |
| 標準仕様 | ラインナップごとに決まるデフォルト構成 |
| 差額 | 標準仕様からの変更による金額増減 |
| 概算原価 | 工種別原価の積み上げ合計 |
| 想定売価 | 概算原価 ÷ (1 - 粗利率) |
| 粗利率 | (売価 - 原価) ÷ 売価 |
