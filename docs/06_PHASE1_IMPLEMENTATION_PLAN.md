# Phase 1 実装計画・優先順位

## Phase 1 の定義

**ゴール：** 現行スプレッドシートと同じ条件を入れたら、同じ工種が出て、同じ金額が出て、どこを人が確認すべきかも分かる状態にする。

**制約：**
- Cloudflare Pages + Workers + D1
- OpenAI APIは条件チェック機能のみPhase1に含む
- 認証は簡易（パスワード認証 or Cloudflare Access）
- 対象ユーザーは社内5〜10名

### Step 4: 原価自動計算エンジン（5〜7日）★最重要

**やること：**
- 計算エンジンのコアロジック実装
  - `POST /api/projects/:id/calculate`
- 計算方式ごとのハンドラー実装
  - fixed_amount
  - per_tsubo
  - per_m2
  - per_meter
  - range_lookup
  - lineup_fixed
  - rule_lookup
  - package_with_delta
  - threshold_surcharge
- 条件ルール評価エンジン
  - cost_rule_conditionsの評価
  - 自動選択/非選択の決定
  - 条件加算の処理
- quantity_rule_tablesの評価（WB部材等）
- lineup_packagesの評価
- project_cost_items の自動生成
- project_cost_summaries の自動集計
- project_warnings の自動生成

**計算フロー：**
```
1. 案件条件を読み込む
2. 全工種をループ
3. 工種内の全明細マスタをループ
4. 各明細に対して:
   a. 条件ルールを評価 → 採用/非採用を決定
   b. 計算方式に応じて数量を算出
   c. 数量 × 単価で金額を算出
   d. project_cost_items に保存
5. 工種ごとの合計を project_cost_summaries に保存
6. 警告条件を評価 → project_warnings に保存
7. 全体合計 + 粗利率を算出
```

**自動化対象（優先度A工種）：** 17工種

| # | 工種 | 主な計算方式 |
|---|------|------------|
| 1 | 木工事 | lineup_fixed / range_lookup |
| 2 | 基礎工事 | per_m2 + range_lookup |
| 3 | 断熱材 | per_m2 + 条件セット |
| 4 | 真壁パネル | per_m2 + per_tsubo |
| 5 | 電気設備工事 | per_tsubo + 条件加算 |
| 6 | 屋根工事 | per_m2 + per_meter |
| 7 | 現場管理費 | per_tsubo |
| 8 | 瑕疵担保保険 | range_lookup |
| 9 | 美装工事 | per_m2 + エリア加算 |
| 10 | 産廃ボックス | package_with_delta |
| 11 | 金物 | per_tsubo |
| 12 | レッカー | fixed_amount |
| 13 | 防蟻工事 | fixed_amount + 条件 |

| 15 | 浄化槽工事 | range_lookup + エリア |
| 16 | 設計業務 | lineup_fixed + fixed |

**成果物：**
- 案件条件入力 → 全工種の原価自動計算が動作
- スプレッドシートと同じ数字が出ることを検証済み

## リスクと対策

### リスク1：スプレッドシートの計算ロジックが一部不明
- **対策：** 不明な箇所は担当者にヒアリング。暫定値で実装し、後で調整。
- **該当：** 足場工事、建材・副資材の詳細が未提出。

### リスク2：マスタデータの網羅性
- **対策：** 最初は主要パターンのみ投入。実運用で不足が出たら順次追加。

### リスク3：条件分岐の複雑さ
- **対策：** Step 4の計算エンジンは十分な時間を確保。テスト駆動で実装。

### リスク4：現場との数字不一致
- **対策：** Step 11で必ずスプレッドシートとの突合を行う。差異は1件ずつ解消。

### リスク5：D1のパフォーマンス
- **対策：** 1案件あたり100〜200行程度なので問題ないはず。JOINは最小限に。
