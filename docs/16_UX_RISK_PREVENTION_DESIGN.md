# 見積ミス防止 UX 設計要件 v1

> **目的**: 本システムの本来目的「見積もりミスを減らす・見落としをなくす・粗利崩れを防ぐ・誰がやっても一定精度で出せる」を UI/UX レベルで実現するための要件を定義する。
> **位置づけ**: 03_SCREEN_DESIGN_v3.md（画面仕様）の上位方針。画面仕様に反映すべき UX 要件の根拠ドキュメント。
> **前提**: 技術的な「壊れにくさ」（13_AI_DEV_TEAM_INSTRUCTIONS.md v3）は確保済み。本ドキュメントは「使い勝手で事故を防ぐ」ための追加レイヤー。

---

## 1. 本来目的と評価軸

### 1-1. 本来目的（再確認）

このシステムは DB を綺麗に作ることが目的ではない。以下を達成することが目的である。

| # | 目的 | 現状の課題 |
|---|------|-----------|
| G-01 | 見積もり間違いをなくす | 手計算・転記でミスが発生 |
| G-02 | 見落としをなくす | 条件・工種の選び忘れ |
| G-03 | 後から原価が膨らんで粗利が崩れるのを防ぐ | 追加工事が後出しで発生 |
| G-04 | 誰がやっても一定精度で出せるようにする | 経験者の勘に依存 |

### 1-2. 追加すべき評価指標

現行の評価指標（migration が通る、seed が入る、snapshot が生成できる、画面が出る）に加え、以下を Phase 1 完了基準に追加する。

| # | 評価指標 | 測定方法 | Phase 1 目標 |
|---|---------|---------|-------------|
| KPI-01 | 見積漏れ警告件数 | project_warnings で warning_type = 条件漏れ系 | テスト案件4件で警告が正しく出る |
| KPI-02 | 手修正率 | 手修正明細数 / 全明細数 | 計測可能な状態にする |
| KPI-03 | 再生成差分の確認漏れ率 | 未承認 diff 件数 / 全 diff 件数 | 差分が全件表示・承認可能 |
| KPI-04 | 商談提示額との乖離件数 | 売価見積 vs 粗利率逆算額 | 乖離 5% 超で警告表示 |
| KPI-05 | AI 警告の採用率 | 解決済み / 全 AI 警告 | Phase 1: 計測可能な状態 |
| KPI-06 | 入力完了率 | 入力済み条件数 / 必須条件数 | ステップ式UIで可視化 |

## 2. 見積もりミスが起きる瞬間の分析

> 計算式の四則演算ミスではなく、以下が実際のミスの原因。

| # | ミス類型 | 発生タイミング | 現行の防止策 | 本システムでの防止策 |
|---|---------|-------------|------------|-------------------|
| E-01 | **入力漏れ** | 案件条件入力時 | なし（経験依存） | ステップ式入力 + 未入力警告 |
| E-02 | **条件の選び忘れ** | 工種生成前 | なし | 生成前チェックリスト + AI条件漏れチェック |
| E-03 | **現場条件の見落とし** | 現地調査後の反映忘れ | なし | 要確認センターで未反映項目表示 |
| E-04 | **標準外仕様の反映漏れ** | 顧客要望の反映時 | 手動管理 | 手修正理由の構造化 + AI分類 |
| E-05 | **過去と違うのに気づかない** | 再生成後 | なし | diff の優先表示（金額インパクト順） |
| E-06 | **手修正理由が曖昧** | 手修正時 | override_reason 入力 | AI 理由分類 + 再発防止ラベル |
| E-07 | **営業提示額とのズレに気づかない** | 売価確定時 | なし | 乖離警告 + AI差異説明 |

## 3. UX要件定義

### UX-01: 見積リスクセンター（案件ごと）

**目的**: 担当者が「どこを確認すれば事故を防げるか」を一目で分かるようにする。

**表示場所**: COST_OVERVIEW（原価一覧）画面の上部、または専用タブ

**表示項目**:

| # | リスク項目 | データソース | 表示条件 |
|---|-----------|------------|---------|
| RC-01 | 未設定条件 | projects の NULL/未入力フィールド | 1件以上 |
| RC-02 | 要確認工種 | project_cost_items.review_status = 'flagged' | 1件以上 |
| RC-03 | 標準外候補 | 手修正ありの明細 | 1件以上 |
| RC-04 | 商談提示額との差 | project_sales_estimates vs 概算合計 | 乖離 > system_settings.売価乖離閾値 |
| RC-05 | 前回再生成との差分 | project_cost_regeneration_diffs (未承認) | 1件以上 |
| RC-06 | AI 検出の注意点 | project_warnings (source='ai', 未解決) | 1件以上 |
| RC-07 | 未レビュー工種 | project_cost_summaries.category 内の未確認数 | 1件以上 |

**UI仕様**:
- 各リスク項目は **件数バッジ** + **色分け**（赤: 重大、黄: 要確認、緑: 正常）
- クリックで該当箇所にジャンプ
- 全件ゼロ → 「✅ 確認完了」バッジ表示

**対応するミス類型**: E-01, E-02, E-03, E-05, E-07

---

### UX-02: ステップ式入力（基本情報画面）

**目的**: 何を入れればよいか迷わない。未入力が一目で分かる。

**現行**: PROJECT_NEW / PROJECT_DETAIL がフラットなフォーム
**改善**: 6ステップに構造化

| Step | 名称 | 入力項目 | 完了条件 |
|------|------|---------|---------|
| 1 | 建物条件 | lineup, roof_shape, stories | lineup 選択済み |
| 2 | 面積・寸法 | tsubo_number, total_floor_area_m2, building_area_m2, roof_area_m2, outer_wall_perimeter_m | tsubo_number > 0 |
| 3 | 地域条件 | is_shizuoka_prefecture, fire_zone_type, has_flood_risk | 全項目選択済み |
| 4 | 設備条件 | has_solar, solar_kw, has_ecocute, has_floor_heating | has_solar 選択済み |
| 5 | 特殊条件 | insulation_grade, has_snow_area, has_special_foundation, customer_name | insulation_grade 選択済み |
| 6 | 生成前チェック | 全条件サマリー表示 + 未入力ハイライト | 全必須項目入力済み |

**UI仕様**:
- ステップインジケーター（1/6 → 2/6 → ...）
- 各ステップの完了状態をアイコンで表示（✅ / ⚠️ / ❌）
- Step 6「生成前チェック」で全条件を一覧表示し、未入力を赤ハイライト
- 「原価生成」ボタンは Step 6 完了後のみ有効化

**対応するミス類型**: E-01, E-02

**実装方針**: PROJECT_NEW と PROJECT_DETAIL で同じステップコンポーネントを共有。PROJECT_DETAIL では全ステップが展開済みで、編集可能。

---

### UX-03: 差分の優先表示（diff 解決UI強化）

**目的**: 差分を見落とさない。金額インパクトが大きいものが先に見える。

**現行**: DIFF_REVIEW 画面で diff_type 7種を一覧表示
**改善**: 優先度ベースのソートと視覚的重み付け

**表示優先順位**:

| 優先度 | 条件 | 表示 |
|--------|------|------|
| 🔴 Critical | is_significant = true AND 確認済みを壊す差分 | 赤背景 + 「確認済み明細が変更されます」警告 |
| 🟠 High | is_significant = true | オレンジ背景 + 金額差表示 |
| 🟡 Medium | amount_changed で差額 > 10,000円 | 黄背景 |
| ⚪ Low | その他の差分 | 通常表示 |

**追加UI要素**:
- **差分サマリーカード**: 「増えた: +3件 / 減った: -1件 / 金額増: +285,000円 / 金額減: -42,000円」
- **インパクト順ソート**: デフォルトは金額インパクト降順
- **カテゴリフィルター**: 差分を diff_type 別にフィルタ可能
- **確認済み破壊の警告**: review_status = 'confirmed' の明細が変更される場合、専用警告

**対応するミス類型**: E-05

---

### UX-04: 手修正理由の構造化

**目的**: 手修正理由が曖昧にならない。後の分析に使える。

**現行**: override_reason はフリーテキスト
**改善**: 理由コード + 自由記述 の二段構成

**理由コード（reason_category）**:

| コード | 表示名 | 説明 | 例 |
|--------|--------|------|-----|
| `site_condition` | 現場条件 | 現地調査で判明した追加要素 | 地盤が悪く基礎補強必要 |
| `customer_request` | 顧客要望 | 施主の仕様変更・追加要望 | 無垢フローリングに変更 |
| `regulatory` | 行政・法規 | 自治体条件・申請費 | 浄化槽設置義務あり |
| `spec_change` | 仕様変更 | 設計変更・グレード変更 | 断熱等級6→7に変更 |
| `price_update` | 単価改定 | 業者単価の変更 | 木材単価値上げ反映 |
| `correction` | 見落とし補正 | 当初の入力・条件漏れ修正 | 足場面積の算出誤り |
| `vendor_quote` | 業者見積反映 | 業者からの正式見積を反映 | サッシ見積 ○○建材 |
| `other` | その他 | 上記に該当しない | — |

**UI仕様**:
- ドロップダウンで理由コード選択（必須）
- テキストエリアで詳細記述（任意だが推奨）
- 理由コードは project_cost_items に `override_reason_category` カラムとして追加（TEXT, nullable）
- Phase 2 で AI による自動分類候補表示

**DB影響**: project_cost_items に `override_reason_category` カラム追加が必要（Phase 1 migration に含めるか、Phase 1.5 で追加するか要判定）

**対応するミス類型**: E-04, E-06

### UX-05: 手修正集計ダッシュボード（管理画面）

**目的**: どこで人が毎回いじっているかを見えるようにし、改善対象を特定する。

**表示場所**: DASHBOARD に管理者向けセクション、または専用管理画面

**表示項目**:

| # | 指標 | 集計方法 | 表示形式 |
|---|------|---------|---------|
| MA-01 | 手修正率が高い工種 TOP 10 | category_code ごとの手修正件数 / 全件数 | 棒グラフ |
| MA-02 | 差額が大きい工種 TOP 10 | \|final_amount - auto_amount\| の平均 | 棒グラフ |
| MA-03 | AI 警告が多い工種 TOP 10 | project_warnings の category_code 集計 | 棒グラフ |
| MA-04 | 手修正理由コード分布 | override_reason_category の件数分布 | 円グラフ |
| MA-05 | 月次の手修正率推移 | 月別の手修正件数 / 全件数 | 折れ線グラフ |

**権限**: admin, manager のみ閲覧可能

**Phase分け**: 
- Phase 1: MA-01, MA-02 のみ（SQLで集計、シンプルテーブル表示）
- Phase 2: MA-03〜MA-05 追加、グラフ表示

**対応するミス類型**: E-04, E-06（組織的な改善）

---

### UX-06: 入力完了率の可視化

**目的**: 案件の入力がどこまで終わっているかを一目で把握。

**表示場所**: DASHBOARD の案件一覧テーブル + PROJECT_DETAIL

**入力完了率の計算**:

```
必須条件 = [lineup, tsubo_number, total_floor_area_m2, insulation_grade, 
            is_shizuoka_prefecture, fire_zone_type, roof_shape]
入力済み = 上記のうち NULL でないもの
完了率 = 入力済み / 必須条件数 × 100%
```

**DASHBOARD での表示**:
- 案件一覧テーブルに「入力率」カラム追加
- プログレスバー表示（赤 < 50% < 黄 < 80% < 緑）

**対応するミス類型**: E-01

---

### UX-07: 生成前チェックリスト

**目的**: 原価生成ボタンを押す前に、条件の最終確認を強制する。

**表示場所**: COST_OVERVIEW の「計算実行」ボタン押下時のモーダル

**チェック項目（自動判定）**:

| # | チェック項目 | 判定条件 | 不合格時の表示 |
|---|------------|---------|-------------|
| PC-01 | 面積入力済み | tsubo_number > 0 | ⚠️ 面積が未入力です |
| PC-02 | ラインナップ選択済み | lineup IS NOT NULL | ❌ ラインナップが未選択です |
| PC-03 | 地域条件入力済み | is_shizuoka_prefecture IS NOT NULL | ⚠️ 地域条件が未入力です |
| PC-04 | 断熱等級入力済み | insulation_grade IS NOT NULL | ⚠️ 断熱等級が未入力です |
| PC-05 | 前回生成からの条件変更 | conditions changed since last snapshot | ℹ️ 前回生成後に条件が変更されています |

**UI仕様**:
- 不合格項目がある場合: 「⚠️ 未入力の条件があります。このまま生成しますか？」確認ダイアログ
- 全合格の場合: 即座に生成ジョブ投入
- ❌（Critical）がある場合: 生成ボタン無効化

**対応するミス類型**: E-01, E-02

## 7. UX 設計原則（AI 駆動開発チーム向け）

> 03_SCREEN_DESIGN_v3.md に従いつつ、以下の原則を追加で遵守すること。

### 原則 1: 「入力する画面」ではなく「見落としを止める画面」

画面の主役はフォームではなく、**警告・リスク・未確認項目**の表示である。フォーム入力は手段であり、目的ではない。

### 原則 2: 「全部見せる」ではなく「危ないものから見せる」

diff も警告も工種一覧も、**金額インパクト・リスク度**でソートする。全部同じ重みで並べると見落とす。

### 原則 3: 「入力完了」を見えるようにする

ユーザーは「何が終わって何が残っているか」を常に把握できるべき。プログレスバー、ステップインジケーター、完了バッジを活用する。

### 原則 4: 「理由」を構造化する

フリーテキストだけでは後の分析に使えない。理由コード + 自由記述 の二段構成にする。

### 原則 5: 「判断支援」であって「判断代行」ではない

システムもAIも「提案・警告」のみ。最終判断は人間が行う。自動反映は禁止（P-10 準拠）。
